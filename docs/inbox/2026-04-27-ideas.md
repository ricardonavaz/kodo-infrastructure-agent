### IDEA #2 — Bugs visuales en rendering de markdown (3 problemas)

**Capturada:** 2026-04-27 (sesion continuacion)

**Contexto:** Tras cerrar arq-19 fix, Ricardo verifico funcionamiento ejecutando comandos en chat principal. La salida del agente revelo 3 bugs distintos en rendering de markdown. Diagnostico tecnico exhaustivo capturado para fix posterior.

**Bug 1 — RESUELTO en esta sesion (arq-13):**

Causa raiz: DataTable.jsx no aplicaba inlineFormat a celdas, mientras formatMessage.jsx y Terminal.jsx local si lo hacen. El semantic parser prefiere convertir tablas markdown a data_table blocks, routeando contenido por el path que NO procesaba inline format.

Fix aplicado: agregado import de inlineFormat + dangerouslySetInnerHTML en headers y celdas de DataTable.jsx.

**Bug 2 — > verde sin contexto (NO LOCALIZADO):**

Sintoma: caracter > aparece visible en color verde en lineas vacias del output del agente, sin texto despues.

Diagnostico de Claude Code:
- Cero handling de blockquote en codigo (formatMessage.jsx y Terminal.jsx)
- Cero CSS para blockquote o md-quote
- inlineFormat no transforma > en ninguna de sus 4 regex

Posibles explicaciones (sin evidencia adicional):
- A) El verde no es del > sino de algo adyacente (emoji, prefijo, marker emitido por agente)
- B) Renderer adicional no encontrado en grep
- C) CSS heredado escondido (archivo CSS tiene 4000+ lineas)

Necesario para fix: screenshot del bug + raw markdown que produjo el agente (texto literal pre-render). Sin evidencia adicional, fix seria adivinar.

Complejidad estimada (si confirma blockquote sin handler): trivial. Riesgo bajo. Tiempo 30-45 min con tests.

**Bug 3 — ** con cierre roto (DIAGNOSTICADO):**

Sintoma: texto en bold con asterisco suelto al final. Ejemplo del screenshot: "POST-INSTALACION.*"

Causa raiz: inlineFormat usa 4 regex con pareo simetrico estricto (lineas 4-10 de utils/formatMessage.jsx, duplicado en Terminal.jsx).

Si Claude emite asimetria (3 asteriscos en lugar de 4 por typo), el regex de bold no matchea y el regex de italic captura el ** como par vacio + el * final queda huerfano.

La causa real es Claude emitiendo markdown malformado, no bug puro de rendering.

Tres opciones de fix:
- (a) Pre-procesar inlineFormat para detectar asteriscos huerfanos (impares) y escaparlos. ~10 lineas. Conservador, predecible.
- (b) Tolerar cierre opcional al final de linea: \\*\\*(.*?)(?:\\*\\*|$). Riesgo: agresivo, puede sobrematchear contenido literal.
- (c) Migrar a parser real tipo marked-mini. Refactor mayor.

Recomendacion: (a). Complejidad media. Riesgo medio. Tiempo 1-2h con tests.

**Deudas tecnicas adicionales descubiertas durante diagnostico:**

1. formatMessage.jsx duplicado en Terminal.jsx (~140 lineas de codigo identico). Cualquier fix de inlineFormat debe aplicarse a AMBOS o consolidar primero.

2. DataTable.jsx no integrado al sistema de markdown rendering del resto. Su fix de Bug 1 lo integra, pero queda decision arquitectural lateral: queremos un unico patron centralizado para que TODOS los componentes que reciben texto de Claude lo procesen markdown automaticamente?

**Estado:** parcial (Bug 1 resuelto, Bug 2 y 3 pendientes para sesion de clasificacion futura)

**Destino propuesto:** input para sesion de clasificacion de los 13 puntos. Bug 2 requiere captura de evidencia adicional antes de fix. Bug 3 requiere decision sobre opcion (a)(b)(c).

**Resumen de las dos capacidades:**

**MODO DIDACTICO**

Modalidad de ejecucion donde KODO no solo ejecuta sino que explica pedagogicamente cada accion. Al activar, cada paso debe documentarse con:
1. Nombre del paso
2. Objetivo tecnico
3. Comando, funcion o accion ejecutada
4. Sintaxis utilizada
5. Resultado obtenido (raw)
6. Interpretacion del resultado
7. Condicion esperada
8. Evaluacion
9. Conclusion parcial
10. Siguiente accion

Al cerrar la ejecucion, KODO genera analisis consolidado que explica que significa el conjunto de resultados, no solo si cada uno fue positivo.

Aplicable a: playbooks de diagnostico, instalacion, reparacion, hardening, monitoreo, comandos individuales, secuencias, actualizaciones, instalacion de software, investigacion de errores, validaciones, revision de logs, optimizacion.

Valor estrategico:
- Para tecnicos: confianza + auditoria
- Para no expertos: aprendizaje progresivo
- Para soporte/documentacion: cada ejecucion = documentacion operativa
- Para auditoria: reconstruccion de razonamiento, no solo resultado
- Para entrenamiento interno KODO: alimenta base de conocimiento

**MODO LEARNOPS (Modo Aprendizaje / Sesion Inteligente)**

Funcion distinta pero complementaria. Registra sesion operativa completa con objetivo declarado por usuario, analiza al finalizar, y convierte experiencia real en playbook optimizado.

Critico cuando una tarea NO sale bien al primer intento. Ejemplo Redmine: instalacion falla, dependencias incompatibles, conflictos versiones, multiples rutas probadas, comandos correctivos, revision logs, ajustes, finalmente exito.

Capacidades:
- Inicio: usuario declara objetivo (ej: "instalar Redmine en Debian 13 con Apache+MariaDB+Passenger")
- Durante: KODO registra TODO (comandos, errores, reintentos, decisiones IA, intervenciones humanas)
- Al finalizar: usuario detiene sesion, KODO genera multiples salidas
- Distincion automatica: KODO debe identificar exploracion vs error vs correccion vs solucion definitiva

Salidas al finalizar:
A. Bitacora completa (cronologico verboso)
B. Playbook optimizado (limpio, ordenado, eficiente)
C. Resumen ejecutivo
D. Diagnostico de lo ocurrido
E. Lista de errores con causas y soluciones
F. Lecciones aprendidas
G. Checklist de validacion
H. Recomendaciones para futuras instalaciones

- apt install ruby-dev

- revision de logs
- ajuste de database.yml
- migracion de base de datos
- configuracion Apache
- reinicio de servicios
- validacion final

Playbook optimizado generado por LearnOps:

2. Actualizar paquetes

LearnOps NO copia mecanicamente. Refina la experiencia real en procedimiento limpio.

**FUNCION AVANZADA — Replay Inteligente:**

### IDEA #11 — Divergencia arquitectural entre path bloques (done event) y path markdown (getHistory) — workaround temporal en Fix C de B3.4

**Capturada:** 2026-04-29 (sesion madrugada ~3:30 AM, durante sprint Bloque 3
sub-paso B3.4)

**Origen:** durante implementacion de B3.4 (pildoras inline para [ACTION:]
markers), Ricardo observo comportamiento erratico:

- Primera vez que el agente respondia con [ACTION:] markers, las pildoras NO
  aparecian
- Botones verdes de "Acciones Sugeridas" tampoco aparecian primera vez
- Tablas markdown se renderizaban rotas (pipes literales | visibles, > entre
  filas)
- Despues de cambiar de servidor y volver, TODO aparecia correctamente
- Hard refresh tambien hacia aparecer todo

Diagnostico tecnico profundo (Claude Code, 7 investigaciones) revelo que NO
era race condition ni cache. Era **bug arquitectural pre-existente desde B3.2**.

**Causa raiz:**

El servidor entrega DOS representaciones distintas del mismo mensaje del agente
dependiendo de como se recibe:

1. **Evento done (primera vez):** mensaje viene con campo `blocks` poblado por
   semantic-parser server-side
```javascript
   case 'done':
     setMessages(prev => [...prev, {
       role: 'assistant',
       content: event.data.response,
       blocks: event.data.blocks || null,  // <-- aqui viene poblado
       ...
     }]);
```

2. **getHistory (al volver del otro servidor):** mensaje viene SIN campo blocks
   o con blocks=null
```javascript
   api.getHistory(connection.id).then(setMessages);  // server NO persiste blocks
```

**Efecto del bug:**

El cliente bifurca completamente el rendering basado en presencia de blocks:

```javascript
// SmartMessage.jsx
if (blocks && blocks.length > 0) {
  // Path BLOQUES: renderiza componentes (DataTable, Recommendation, etc)
  // formatMessage NO se llama
  // Pills [ACTION:] NO aparecen
  // Botones verdes suprimidos por filtro !msg.blocks en Terminal.jsx:461
} else {
  // Path MARKDOWN: formatMessage(content) procesa via marked + DOMPurify
  // Pills SI aparecen
  // Botones verdes SI aparecen
}
```

Por eso:
- Primera vez (done con blocks) -> path bloques -> pildoras y botones suprimidos
- Despues de cambiar y volver (history sin blocks) -> path markdown -> todo aparece

**Workaround aplicado (Fix C en commit de B3.4):**

En Terminal.jsx case 'done' del handleStreamEvent, forzar blocks=null:

```javascript
case 'done':
  setMessages(prev => [...prev, {
    role: 'assistant',
    content: event.data.response,
    // FIX C (B3.4): Forzar blocks=null para consistencia con history.
    blocks: null,
    tags: event.data.tags || [],
    metrics: event.data.metrics,
  }]);
```

Resultado: cliente siempre usa path markdown. Comportamiento consistente entre
done y history. Pildoras, botones verdes y tablas funcionan primera vez.

**Costo del workaround:**

Las cards visualmente bonitas que el path bloques renderizaba (Recommendation
con borde naranja, DataTable, MEDIA tags) ahora se renderean como markdown
puro. La perdida visual es menor porque:
- El path bloques estaba subdesarrollado (no soportaba pills, suprimia botones)
- El markdown puro es perfectamente legible para el usuario
- Recommendations con tablas + bullets se ven bien via markdown

**Decision arquitectural mayor (DIFERIDA - no resolver ahora):**

El bug expone tension fundamental entre dos modos de rendering. La decision
correcta requiere construir P2 expandido (Bloque 4 del plan, 50-70h estimadas):

**Opcion A — Path bloques completo (path elegido en plan original):**
- Server emite blocks ricos
- Cliente renderiza componentes especializados (DataTable, Recommendation,
  Action, Finding, etc)
- Cards bonitas, jerarquia visual clara, interaccion rica
- Persiste blocks en DB para consistencia con history
- Requiere completar todos los componentes y soportar pills/acciones dentro
  de ellos

**Opcion B — Path markdown solo (workaround actual):**
- Server emite markdown crudo
- Cliente procesa via formatMessage (marked + DOMPurify)
- UI mas simple, soporta pills [ACTION:] inline naturalmente
- Pierde diferenciacion visual entre tipos de contenido
- Sin componentes especializados ni interactividad rica

**Opcion C — Hibrido:**
- Server emite ambos: markdown + blocks
- Cliente decide cual usar segun contexto
- Mas complejo pero permite migrar gradualmente

**Recomendacion sin compromiso:** Opcion A es el camino largo que respeta el
diseño original del plan. Pero requiere trabajo significativo (incluido en
Bloque 4: P2 expandido completo, 50-70h). Hasta entonces, Fix C mantiene UX
funcional.

**Conexiones arquitecturales:**

1. **Bloque 4 (P2 expandido completo):** este bug debe resolverse
   definitivamente cuando se construya P2. El path bloques completo es parte
   del scope de P2. Cuando se construya, REVERTIR Fix C en Terminal.jsx.

2. **D1 del documento de clasificacion (P2 expandido 50-70h):** confirma que
   el path bloques es trabajo arquitectural mayor. Decision D1 ya aprobada.

3. **Bloque 10 (Sistema de Diseño):** las cards de path bloques son parte del
   sistema de diseño formal. Sin sistema de diseño completo, las cards parciales
   se ven inconsistentes con el resto de la UI.

4. **D3 del documento de clasificacion (Sistema de Diseño antes de P3+):**
   refuerza que cards y componentes especializados deben venir despues del
   sistema de diseño formal.

5. **B11/B12/B13 del IDEA #5 (bugs visuales):** algunos de esos bugs viven en
   el path bloques. Se resolveran junto con P2 expandido.

**Pregunta diferida sobre semantic-parser server-side:**

El semantic-parser que genera blocks en el server existe pero produce output
inconsistente (a veces tablas malformadas, blocks parciales). Cuando se
construya P2 expandido habra que:
- Auditar el semantic-parser
- Decidir si se mantiene, refactoriza o reemplaza
- Definir contrato exacto de blocks (schema)
- Persistir blocks en DB con migracion

**Estado:**

- Workaround Fix C aplicado y funcional
- B3.4 + Fix C cierran el sprint Bloque 3 (sub-pasos B3.1-B3.5 completos)
- Decision arquitectural diferida hasta sesion dedicada antes de Bloque 4

**Lecciones de proceso:**

- Bug "erratico" llevo a diagnostico tecnico profundo (7 investigaciones) que
  revelo causa arquitectural, no bug puntual
- Decisiones arquitecturales tomadas hace tiempo (semantic-parser, schema de

- Workarounds bien documentados son aceptables como puente hacia solucion

**Recordatorio para futura sesion:**

Cuando se inicie Bloque 4 (P2 expandido completo):

3. Si A elegido: revertir Fix C en Terminal.jsx + completar componentes path
